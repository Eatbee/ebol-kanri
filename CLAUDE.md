# EBOL英会話 管理システム

## プロジェクト概要
英会話スクール向けのレッスン管理アプリ。先生がレッスン報告を入力し、管理者が実績・予約を管理する。

| 項目 | 内容 |
|------|------|
| 技術スタック | Python / Streamlit / Supabase |
| デプロイ先 | Streamlit Community Cloud（streamlit.app） |
| GitHub | https://github.com/Eatbee/ebol-kanri |
| ローカル起動 | `streamlit run lesson_report_app.py` |

## 主要ファイル
- `lesson_report_app.py` — エントリーポイント・PIN認証・ナビゲーション
- `utils.py` — Supabase連携・共通ユーティリティ
- `pages/01_先生_報告フォーム.py` — レッスン報告フォーム
- `pages/02_実績一覧.py` — レッスン実績一覧
- `pages/03_予約管理.py` — 予約管理フォーム
- `pages/04_月次一覧表.py` — 月次集計表
- `pages/admin_main.py` / `admin_login.py` — 管理者ページ
- `.streamlit/secrets.toml` — Supabase接続情報・PINコード（ローカル専用）

## 作業ログの運用ルール

### セッション開始時に必ずやること
`.claude/worklog.md` が存在する場合は最初に読み、前回の作業内容・未完了タスクを把握してからユーザーに話しかける。

### 作業ログを更新するタイミング
以下のいずれかが起きたら、`.claude/worklog.md` を自動的に更新すること（ユーザーへの確認は不要）:

1. ユーザーが **「ここまで」** と入力した
2. 実装・修正・設計判断が一区切りついた
3. `git commit` が完了した
4. エラーの原因究明や重要な発見があった

### worklog.md の記述フォーマット
**既存の内容は消さず、先頭に追記**する（新しい記録が上に来る）。

```
---
## YYYY-MM-DD（例: 2026-03-02）

### やったこと
-

### 変更したファイル
- ファイル名：変更内容

### 設計判断・はまったこと
-

### 次回やること・未完了
-
---
```

## 開発上の注意
- **デプロイ反映** には git push が必要（変更がローカルのままだと streamlit.app に反映されない）
- **push 前に必ず確認**する（ユーザーの許可を得てから push）
- コミットはユーザーに明示的に依頼された場合のみ行う
- やりとりは日本語で行う
